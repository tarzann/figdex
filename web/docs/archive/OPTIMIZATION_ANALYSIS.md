# ניתוח אופטימיזציה - פניות לשרתים

## מצב נוכחי:

### Client-Side Polling:
1. **`checkJobStatuses`** - כל 15 שניות
   - קריאה ל-Vercel API (`/api/get-job-status`)
   - Vercel קורא ל-Supabase (1 query)
   - **תוצאה:** ~4 פניות לדקה ל-Vercel, ~4 פניות לדקה ל-Supabase

2. **`processIndexJob`** - כל 10 שניות (כשיש job פעיל)
   - קריאה ל-Vercel API (`/api/process-index-job`)
   - Vercel קורא ל-Supabase (טעינת job + עדכון job)
   - Vercel קורא ל-Figma API (לכל chunk של frames)
   - **תוצאה:** ~6 פניות לדקה ל-Vercel, ~12 פניות לדקה ל-Supabase, ~6 פניות לדקה ל-Figma API

3. **`updateIndexInAPI`** - בכל פעם שיש שינוי ב-status
   - קריאה ל-Vercel API (`/api/saved-indices`)
   - Vercel קורא ל-Supabase (1 update query)
   - **תוצאה:** תלוי במספר השינויים, יכול להיות 1-10 פניות לדקה

### סה"כ פניות לדקה (עבור job אחד פעיל):
- **Vercel:** ~10-14 פניות/דקה
- **Supabase:** ~16-26 פניות/דקה  
- **Figma API:** ~6 פניות/דקה

## בעיות:

1. **כפילות:** גם `checkJobStatuses` וגם `processIndexJob` קוראים לשרת
2. **Polling מיותר:** גם כשאין שינויים
3. **עדכונים מיותרים:** `updateIndexInAPI` נקרא גם על שינויים קטנים
4. **מרווחים קצרים:** 10-15 שניות זה עדיין תכוף

## אופטימיזציות מומלצות:

### 1. הגדלת מרווחי Polling (קל ליישום, הפחתה של 50-60%)
- `checkJobStatuses`: 15s → 30s (2 פניות/דקה במקום 4)
- `processIndexJob`: 10s → 20s (3 פניות/דקה במקום 6)
- **חיסכון:** ~50% פחות פניות

### 2. Adaptive Polling (הפחתה של 70-80%)
- התחלה: 10 שניות
- אם אין שינויים: הגדל ל-30 שניות
- אם יש שינויים: חזור ל-10 שניות
- **חיסכון:** ~70% פחות פניות כשאין פעילות

### 3. איחוד קריאות (הפחתה של 30-40%)
- לשלב `checkJobStatuses` ו-`processIndexJob` לקריאה אחת
- `processIndexJob` יכול להחזיר גם את ה-status
- **חיסכון:** ~30% פחות פניות

### 4. הפחתת עדכוני DB (הפחתה של 20-30%)
- לעדכן `updateIndexInAPI` רק אם יש שינוי משמעותי (status, progress > 5%)
- לא לעדכן על כל שינוי קטן
- **חיסכון:** ~20% פחות עדכונים

### 5. Debouncing של עדכונים (הפחתה של 10-20%)
- לאסוף עדכונים ולשלוח אותם יחד כל 5 שניות
- במקום עדכון מיידי על כל שינוי

## המלצה:

**שלב 1 (קל, חיסכון של 50%):**
- הגדל מרווחים: 30s ל-status check, 20s ל-processing
- הפחת עדכוני DB: רק על שינויים משמעותיים

**שלב 2 (בינוני, חיסכון נוסף של 30%):**
- Adaptive polling
- Debouncing של עדכונים

**שלב 3 (מתקדם, חיסכון נוסף של 20%):**
- איחוד קריאות
- WebSockets (אם יש תקציב)

## יתרונות:

1. **עלויות נמוכות יותר:**
   - פחות קריאות = פחות עלויות Vercel/Supabase
   - פחות קריאות ל-Figma API = פחות rate limiting

2. **ביצועים טובים יותר:**
   - פחות עומס על השרתים
   - פחות latency
   - פחות שגיאות timeout

3. **חוויית משתמש:**
   - עדיין responsive (30s זה עדיין מהיר)
   - פחות שגיאות network
   - יותר יציב

## סיכון:

- Polling איטי יותר = עדכונים פחות תכופים
- אבל 30 שניות זה עדיין סביר לרוב המקרים
