# FigDex Activation Plan

**Last updated:** March 30, 2026

This document translates current UX, marketing, and product-friction findings into a practical plan for improving user activation before launch.

## 1. Activation Goal

The target activation journey is:

1. user understands what FigDex does
2. user installs the plugin
3. user connects or starts as guest
4. user links a Figma file
5. user creates a first index
6. user opens the result in FigDex Web
7. user understands how to search and share

If this path feels obvious and rewarding, the product is ready for early market use.

## 2. Activation Success Criteria

The activation flow is considered healthy when a first-time user can:
- understand the product in under a minute
- reach installation without confusion
- create a first index without asking what happens next
- land in a gallery that feels understandable
- understand the difference between free, guest, and upgraded use

## 3. Workstreams

## 3.1 Workstream A — Core Messaging

### Goal

Create one consistent product story across landing, plugin, pricing, account, and download.

### Required outputs

- one category statement
- one plugin/web relationship statement
- one ICP statement
- one upgrade value framework

### Proposed baseline copy direction

Category statement:
- FigDex turns large Figma files into a browsable, searchable design library.

Plugin/web relationship:
- Use the plugin to capture and update designs from Figma. Use FigDex Web to browse, search, review, and share them.

ICP:
- For design teams, design ops, and product organizations working across large Figma files.

### Tasks

1. Rewrite landing hero and support sections around outcomes.
2. Rewrite plugin download page to focus on first success, not package mechanics.
3. Rewrite pricing section intros around when each plan is needed.
4. Clean old brand terms and technical-first copy from login/account/onboarding surfaces.

## 3.2 Workstream B — First-Time Path

### Goal

Make the first-time journey feel linear and low-risk.

### Desired path

Landing -> Download Plugin -> Install -> Connect/Guest -> Link File -> First Index -> Open Result

### Tasks

1. Make homepage CTA hierarchy support one main path.
2. Make download page the clearest installation step in the product.
3. Make plugin onboarding explain only the next action, not the whole system.
4. After first signup/login, route the user toward first index success, not a generic empty system surface.

## 3.3 Workstream C — Plugin Activation

### Goal

Reduce mental load inside the plugin during the first session.

### Problems to solve

- too many state transitions
- too much conceptual understanding required
- unclear relationship between guest, free, and connected states

### Tasks

1. Audit every plugin state and remove redundant copy.
2. Ensure each plugin screen has one dominant CTA.
3. Clarify what guest mode means and what changes after connecting.
4. Ensure limit states move the user forward clearly.

## 3.4 Workstream D — Gallery First Session

### Goal

Make the gallery understandable immediately after first indexing.

### Problems to solve

- too much navigation and control before confidence exists
- new users land in a powerful but dense environment

### Tasks

1. Define a new-user gallery mode or reduced first-session presentation.
2. Add context guidance:
   - what the user is looking at
   - what to do next
   - how search/share fit in
3. Delay advanced filters and controls until after first successful exploration.

## 3.5 Workstream E — Share Simplification

### Goal

Make sharing understandable in human terms.

### Tasks

1. Reframe share options around recipient scenarios.
2. Separate share creation from share management where needed.
3. Ensure every share screen answers:
   - what am I sharing
   - who is it for
   - what will they see

## 3.6 Workstream F — Upgrade Path

### Goal

Make upgrading feel like a continuation of success, not punishment.

### Tasks

1. Reframe pricing around who each plan is for.
2. Improve upgrade language in plugin and web.
3. Make limit messaging explain what becomes possible after upgrade.
4. Keep operational limits in backend, but keep user-facing plan explanation simple.

## 4. Screen-by-Screen Priorities

## 4.1 Homepage

### Objective

Get users into the correct first path fast.

### Priorities

1. Reduce competing CTAs.
2. Sharpen the hero around one outcome.
3. Make plugin/web relationship explicit.
4. Improve proof and credibility.

## 4.2 Download Plugin

### Objective

Turn installation into a confident next step.

### Priorities

1. Keep the page outcome-led.
2. Make install steps feel simple and safe.
3. Clarify what happens after install.

## 4.3 Plugin

### Objective

Get the user to first index with minimal friction.

### Priorities

1. One primary next step at all times.
2. Less duplicate explanation.
3. Clear guest/free/connected logic.

## 4.4 Gallery

### Objective

Turn first indexed content into first real value.

### Priorities

1. Reduce first-session density.
2. Add orientation and next-step guidance.
3. Make search and share easier to understand.

## 4.5 Pricing

### Objective

Help users self-select correctly.

### Priorities

1. Clarify who each plan is for.
2. Explain why upgrade matters.
3. Reduce plan confusion.

## 4.6 Account

### Objective

Feel like control and trust, not admin setup.

### Priorities

1. De-emphasize technical actions.
2. Highlight plan value and usage clearly.
3. Keep advanced controls secondary.

## 5. Execution Order

### Phase 1

- messaging system
- homepage CTA cleanup
- download page refinement
- plugin first-session copy cleanup

### Phase 2

- gallery first-session simplification
- share UX simplification
- pricing upgrade clarity

### Phase 3

- account trust/control polish
- admin clarity improvements
- supporting empty/error/loading state pass

## 6. What Not to Do

Do not:
- start with cosmetic polish only
- add more features before activation is clearer
- expose more internal terminology to first-time users
- solve activation with more screens or more explanations

## 7. Immediate Next Step

The next implementation cycle should begin with:

1. defining final core messaging
2. simplifying the first-time path from homepage to first index
3. reducing new-user complexity in the gallery

This should be executed before broader visual polish work.
